Project Summary: Work Tracker

This document provides a detailed and meticulous overview of the Work Tracker Android application, covering its structure, coding practices, and data management.

1) Complete App Structure

a) Brief App Route Map

The Work Tracker application allows users to log work activities across different categories, manage details for these activities (like description, operator, expenses), view activity history, generate reports (including SMS and PDF exports), and manage application preferences (like passwords, API keys, and operator information).

The app starts with a MainActivity that sets up the primary navigation. The main UI is encapsulated within a MainScreen which uses a bottom navigation bar to switch between major sections: Dashboard, History, Reports, ChatBot (placeholder), and Preferences. From the Dashboard or History, users can navigate to a dedicated flow for logging new work activities, starting with category selection (LogWorkActivityScreen) and then proceeding to detail entry (WorkDetailsScreen). A separate ViewLogsScreen is also available for a tabular view of logs, filterable by operator ID.

b) App Navigation Routes

The app utilizes Jetpack Navigation Compose with two main NavHosts:

1. Top-Level Navigation (in MainActivity):

  • AppRoutes.MAIN_SCREEN ("mainScreen"): Start Destination. Loads MainScreen.
  • AppRoutes.LOG_WORK_ACTIVITY ("logWorkActivity"): Navigates to LogWorkActivityScreen.
    • Triggered from DashboardScreen or HistoryScreen.
  • ROUTE_WORK_DETAILS ("work_details/{categoryName}"): Navigates to WorkDetailsRoute (which then displays WorkDetailsScreen), passing categoryName as an argument.
    • Triggered from LogWorkActivityScreen after category selection.
  • AppRoutes.VIEW_LOGS ("viewLogs"): Navigates to ViewLogsScreen. (Currently, no direct UI element seems to navigate here, it might be for future use or internal navigation).

2. Bottom Navigation (within MainScreen):

  • BottomNavScreen.Dashboard.route ("dashboard"): Start Destination for MainScreen. Loads DashboardScreen.
  • BottomNavScreen.History.route ("history"): Loads HistoryScreen.
  • BottomNavScreen.Reports.route ("reports"): Loads ReportsHubScreen.
  • BottomNavScreen.ChatBot.route ("chatbot"): Loads a placeholder Text composable.
  • BottomNavScreen.Preferences.route ("preferences_route"): Loads PreferencesScreen.

c) Details of Various Screens

Top-Level Navigable Screens (from MainActivity):

  1. MainScreen (com.example.worktracker.ui.MainScreen.kt)
    • Purpose: Acts as the primary container for the app's main sections, utilizing a bottom navigation bar.
    • UI: Scaffold with a NavigationBar hosting DashboardScreen, HistoryScreen, ReportsHubScreen, ChatBotScreen (placeholder), and PreferencesScreen.
    • Navigation: Manages navigation between these five sections using its own NavHostController. Passes the MainActivity's NavController to child screens that need to trigger top-level navigation (e.g., to LogWorkActivityScreen).
  2. LogWorkActivityScreen (com.example.worktracker.ui.logwork.LogWorkActivityScreen.kt)
    • Purpose: Allows users to select a category for the work activity they intend to log.
    • UI: Scaffold with a TopAppBar. Displays a LazyColumn of Buttons, each representing an activity category. Buttons for ongoing activities are highlighted (e.g., green). Shows a CircularProgressIndicator if loading or a "No activity categories available" message.
    • Interaction: Selecting a category navigates to WorkDetailsScreen with the chosen category name.
    • ViewModel: LogWorkViewModel. UI State: LogWorkUiState (contains isLoading, categories).
  3. WorkDetailsRoute & WorkDetailsScreen
    • WorkDetailsRoute (com.example.worktracker.ui.screens.workdetails.WorkDetailsRoute.kt):
      • Purpose: Composable responsible for setting up the WorkDetailsScreen and providing it with the necessary ViewModel (GenericWorkDetailsViewModel), state, and actions via a WorkDetailsCoordinator. It extracts the categoryName from navigation arguments.
    • WorkDetailsScreen (com.example.worktracker.ui.screens.workdetails.WorkDetailsScreen.kt):
      • Purpose: Enables users to input and manage the details of a specific work activity. It handles starting, tracking, and ending an activity.
      • UI: Scaffold with a TopAppBar showing the category name and a back button. Contains: Date display; Time information section ("Start" button, start time, end time, live duration); Input fields ("Description", "Operator ID", "Expenses"); Radio button groups ("Task Successful?", "Assigned By:"); "End Activity & Save" button. Displays errors and loading indicators.
      • ViewModel: GenericWorkDetailsViewModel. UI State: WorkDetailsState. Actions: WorkDetailsActions.
  4. ViewLogsScreen (com.example.worktracker.ui.viewlogs.ViewLogsScreen.kt)
    • Purpose: Provides a tabular display of all work activity logs, with an option to filter by Operator ID.
    • UI: Scaffold with a TopAppBar and back button. Contains: An OutlinedTextField for "Filter by Operator ID"; A LogsTable (using LazyColumn) with columns: "Sr.No.", "Date", "Activity", "Duration", "Op.ID". Shows loading indicator or "No logs found..." message.
    • ViewModel: ViewLogsViewModel. UI State: ViewLogsUiState.

Screens within MainScreen (Bottom Navigation):

  1. DashboardScreen (com.example.worktracker.ui.screens.dashboard.DashboardScreen.kt)
    • Purpose: The landing screen. Provides a quick overview of ongoing/recent activities and a shortcut to log new work.
    • UI: If empty, shows welcome message and "Log Your First Activity" button. Otherwise, "Log New Activity" button, lists of "Ongoing Activities" and "Recent Work" (using WorkLogDashboardItem cards).
    • Interaction: "Log" buttons navigate to LogWorkActivityScreen.
    • ViewModel: DashboardViewModel. UI State: DashboardUiState.
  2. HistoryScreen (com.example.worktracker.ui.screens.history.HistoryScreen.kt)
    • Purpose: Displays a comprehensive list of all past work activity logs.
    • UI: If empty, shows message and "Log Your First Activity" button. Otherwise, a LazyColumn of WorkLogItem cards.
    • Interaction: "Log" button navigates to LogWorkActivityScreen.
    • ViewModel: HistoryViewModel. UI State: HistoryUiState.
  3. ReportsHubScreen (com.example.worktracker.ui.reports.ReportsHubScreen.kt)
    • Purpose: Central hub for generating reports and performing analyses.
    • UI: ModalNavigationDrawer (drawer for recent logs for PDF export via ReportsDrawerContent and ExportConfirmationDialog). Main screen has ActionCards for "Send Activity Details (Text)" (uses SelectLogForSmsDialog), and placeholders for "Analyze Expenses", "Analyze Operator Activity vs Free Time", "Analyze Misc Prompt".
    • Interaction: Handles PDF export, SMS intent generation.
    • ViewModel: ReportsViewModel. UI State: ReportsUiState. Event Flow: PdfExportEvent.
  4. ChatBot Screen (Placeholder in com.example.worktracker.ui.MainScreen.kt)
    • Purpose: Intended for a chatbot feature.
    • UI: Currently displays Text("ChatBot Screen (Coming Soon!)").
  5. PreferencesScreen (com.example.worktracker.ui.screens.preferences.PreferencesScreen.kt)
    • Purpose: Allows users to manage application-wide settings.
    • UI: Sections/buttons for: Master Reset Password (SetPasswordDialog, ConfirmActionPasswordDialog); Preferred SMS Contact (SetSmsContactDialog); Gemini API Key (SetGeminiApiKeyDialog); Operator Management (unlocks with OperatorPasswordDialog to show OperatorListDialog for Add/Edit/Delete using AddOperatorDialog, EditOperatorDialog, AlertDialog); Master Reset button.
    • ViewModel: PreferencesViewModel. UI State: PreferencesUiState.

e) Any other details that might help with understanding the App

  • Kotlin & Jetpack Compose: The app is built entirely using Kotlin and Jetpack Compose for the UI.
  • Material Design 3: Utilizes Material 3 components and theming.
  • Hilt: Used for dependency injection throughout the app.
  • Coroutines & Flow: Asynchronous operations and state management heavily rely on Kotlin Coroutines and StateFlow/Flow.
  • Single Activity Architecture: The app uses a single MainActivity which hosts various composable screens managed by Jetpack Navigation.
  • ViewModel per Screen/Feature: Each major screen or feature has its own ViewModel to manage its state and business logic.
  • Centralized SharedPreferences: Settings like master password, SMS contact, and Gemini API key are stored in SharedPreferences, managed via AppModule.kt.

2) Coding conventions and techniques used

a) List of all the good practises used in the architecture

  • MVVM (Model-View-ViewModel) Architecture: Clear separation of UI (Composable Screens), State/Logic (ViewModels), and Data (Repositories).
  • Dependency Injection with Hilt: Simplifies dependency management and improves testability.
  • Repository Pattern: Abstracts data sources (Room DB, SharedPreferences) from ViewModels.
  • Single Source of Truth: ViewModels expose UI state via StateFlow, which is the single source of truth for the UI.
  • Immutability for UI State: UI state objects (e.g., LogWorkUiState, WorkDetailsState) are typically data classes, promoting unidirectional data flow.
  • Use of Kotlin Coroutines and Flow: For efficient asynchronous programming and reactive updates.
  • Jetpack Compose for UI: Modern, declarative UI toolkit.
  • Jetpack Navigation Compose: For structured and type-safe navigation.
  • Room for Local Persistence: Robust and type-safe database interaction.
  • Sealed Classes for Events/States: Used for defining navigation routes (BottomNavScreen) and ViewModel events (e.g., PdfExportEvent).
  • Modular UI Components: Screens are broken down into smaller, reusable composables (e.g., WorkLogItem, dialogs in ui.components).
  • Centralized Dependency Versioning: Use of libs.versions.toml (Gradle Version Catalog).
  • Clear Package Structure: Code is organized by feature (e.g., logwork, reports, preferences) and layer (e.g., data, ui, di).

b) Details about things like Hilt DI etc. being used.

  • Hilt Setup:
    • @HiltAndroidApp on MyApplication.kt to initialize Hilt.
    • @AndroidEntryPoint on MainActivity to enable field injection.
    • @HiltViewModel on all ViewModel classes to enable their creation and injection.
  • Modules:
    • AppModule.kt (@Module, @InstallIn(SingletonComponent::class)) is the primary Hilt module.
    • It provides singletons for SharedPreferences, AppDatabase, WorkActivityDao, OperatorInfoDao, WorkActivityRepository, OperatorRepository, and GeminiProService.
  • Injection: Dependencies are constructor-injected (e.g., DAOs into Repositories, Repositories/SharedPreferences into ViewModels).

c) Details about the UI (Mention what libraries and practises are used)

  • Core UI Library: Jetpack Compose.
  • UI Design System: Material Design 3. Evident from use of MaterialTheme and components like Scaffold, TopAppBar, NavigationBar, Button, Card, AlertDialog, etc.
  • Layout Composables: Column, Row, Box, Spacer, LazyColumn.
  • Modifiers: Extensively used for styling, sizing, padding, arrangement, and event handling.
  • State Hoisting: UI state owned by ViewModels, passed to Composables, which emit events upwards. collectAsState() used to observe StateFlow.
  • Animation: AnimatedVisibility, fadeIn, fadeOut for screen transitions and conditional UI.
  • Navigation: Jetpack Navigation Compose with NavHostController.
  • Dialogs: Standard AlertDialog and custom Composable dialogs, managed via ViewModel UI states. Many custom dialogs in com.example.worktracker.ui.components.
  • Theming: Centralized in com.example.worktracker.ui.theme (Theme.kt, Color.kt, Type.kt).

d) Details of Databases used to store user input. (Explain in detail)

Two main forms of local data storage are used:

1. Room Persistence Library (for structured data):

  • Database Class: AppDatabase.kt (@Database)
    • Version: 4
    • Entities: WorkActivityLog and OperatorInfo.
    • Provides WorkActivityDao and OperatorInfoDao.
    • Includes MIGRATION_2_3 for schema changes to work_activity_logs.
    • Uses fallbackToDestructiveMigration() for other version changes.
  • Entities:
    • WorkActivityLog.kt (@Entity(tableName = "work_activity_logs")): Stores individual work log entries. Fields: id (PK, auto-generated Long), categoryName (String), startTime (Long), endTime (Long?), description (String), operatorId (Int), expenses (Double), logDate (Long), taskSuccessful (Boolean?), assignedBy (String?).
    • OperatorInfo.kt (@Entity(tableName = "operator_info")): Stores information about operators. Fields: operatorId (PK, Int, user-defined), name (String), hourlySalary (Double), role (String), priority (Int), notes (String?), notesForAi (String?).
  • DAOs (Data Access Objects):
    • WorkActivityDao.kt: Interface with methods to insertLog, getAllLogs, getLogById, deleteLogById, clearAllLogsInternal, resetAutoIncrementForLogsTable, wipeDatabaseAndResetIds (transactional), getOngoingActivityByCategoryName, getOngoingActivities, getRecentLogs. Uses Flow.
    • OperatorInfoDao.kt: Interface with methods to insert, update, delete operator, getAllOperators, getOperatorById. Uses Flow.
  • Repositories:
    • WorkActivityRepositoryImpl.kt: Implements WorkActivityRepository. Uses WorkActivityDao. clearAllLogs calls transactional wipeDatabaseAndResetIds.
    • OperatorRepository.kt: Uses OperatorInfoDao.

2. SharedPreferences (for simple key-value settings):

  • Managed in: AppModule.kt.
  • File Name: "WorkTrackerUserPrefs".
  • Purpose: Stores user-specific settings.
  • Keys Used: KEY_MASTER_PASSWORD, KEY_SMS_CONTACT, KEY_GEMINI_API_KEY.
